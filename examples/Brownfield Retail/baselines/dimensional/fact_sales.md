# fact_sales

Baseline documentation for the existing `fact_sales` table in the Snowflake data warehouse. This is the central fact table in the retail sales star schema, recording one row per sales transaction. It has been in production since 2021 and is the primary input to the Retail Sales Analytics dashboard.

## Source DDL

```sql
CREATE TABLE analytics.fact_sales (
    sale_id           BIGINT        NOT NULL PRIMARY KEY,
    sale_date         DATE          NOT NULL,
    store_key         BIGINT        NOT NULL REFERENCES analytics.dim_store(store_key),
    product_key       BIGINT        NOT NULL REFERENCES analytics.dim_product(product_key),
    quantity          INTEGER       NOT NULL,
    unit_price        DECIMAL(10,2) NOT NULL,
    total_amount      DECIMAL(12,2) NOT NULL,
    discount_amount   DECIMAL(10,2),
    payment_method    VARCHAR(20)   NOT NULL,
    customer_id       VARCHAR(50),
    etl_batch_id      BIGINT        NOT NULL,
    dw_load_timestamp TIMESTAMP_NTZ NOT NULL DEFAULT CURRENT_TIMESTAMP(),
    dw_update_timestamp TIMESTAMP_NTZ NOT NULL DEFAULT CURRENT_TIMESTAMP()
);
```

## Column Summary

Column | Type | Nullable | Description
--- | --- | --- | ---
sale_id | bigint | no | Primary key — surrogate key generated by ETL
sale_date | date | no | Date the sale was completed
store_key | bigint | no | FK to dim_store
product_key | bigint | no | FK to dim_product
quantity | integer | no | Number of units sold
unit_price | decimal(10,2) | no | Price per unit at time of sale
total_amount | decimal(12,2) | no | Total transaction amount after discounts
discount_amount | decimal(10,2) | yes | Discount applied to the transaction
payment_method | varchar(20) | no | Payment method used (Cash, Credit, Debit, Mobile)
customer_id | varchar(50) | yes | Loyalty program customer ID; null for non-members
etl_batch_id | bigint | no | ETL batch identifier for lineage
dw_load_timestamp | timestamp_ntz | no | When this row was first loaded
dw_update_timestamp | timestamp_ntz | no | When this row was last updated

## Notes

- **Grain**: One row per sales transaction (not per line item). Multi-product transactions are aggregated at the POS before loading. `product_key` represents the *primary* product in the basket.
- **Dimensions**: dim_store (store_key), dim_product (product_key), dim_date (sale_date)
- **Load**: Daily, SLA available by 06:00 UTC
- **Data quality**: Approximately 2% of transactions have `customer_id` populated for non-loyalty purchases due to a POS bug fixed in 2023-Q3. These records have `customer_id` values that do not match any loyalty account.
- **Discount logic**: `discount_amount` is null (not zero) when no discount was applied. Downstream reports treat null as zero.
