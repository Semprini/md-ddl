## Executive Summary

MD-DDL 0.9.0 is a structurally coherent, well-designed standard with a multi-agent architecture that covers most of the data management lifecycle. The review identified **4 critical issues**, **5 structural weaknesses**, **6 missing capabilities**, **4 specification contradictions**, and **5 agent failure modes** â€” plus a production-blocking gap in **product lifecycle and versioning** that prevents reliable use in organisations with active downstream consumers.

**Post-review actions applied (2026-03-13):** CLA-1 resolved (pii_fields standardised in spec); CLA-2 downgraded to advisory (naming is user choice, not a spec violation); CLA-4 partially resolved (Agent Architect fallback added); self-referential relationship pattern added to Financial Crime example; product lifecycle plan created (`roadmap/v1/plan-productLifecycle.prompt.md`).

### Strongest elements

- Agent Governance (compliance audit, remediation, multi-jurisdiction handling)
- Agent Artifact (physical generation, product-scoped output, reconciliation)
- Brownfield adoption path (schema import, maturity model, coexistence design)

### Weakest elements

- Product lifecycle and versioning (spec-level gap â€” plan created, implementation pending)
- Model evolution and impact propagation (no changelog, no downstream notification)
- Healthcare and non-financial-services industry support

**Top priority for 1.0:** Define product versioning and lifecycle states, specify self-referential relationship patterns in the spec, and complete the architecture skill.

## Cross-Layer Agreement

*Issues identified by 2 or more layers. Highest confidence.*

### CLA-1: `pii_fields` field used in agent prompts but absent from spec schema

- **Found by:** Layer 2 (C1, F3), Layer 3 (R4 failure, compliance audit false positives)
- **Severity:** ~~Critical~~ **Resolved** â€” spec updated 2026-03-13
- **Evidence:**
  - L2: `agents/agent-governance/skills/compliance-audit/SKILL.md` lines 165, 189, 282, 350, 365, 374 require `pii_fields`; spec Section 3 governance metadata schema defined only `pii` (boolean), not `pii_fields`
  - L2: GDPR, HIPAA, CCPA regulator files all require `pii_fields` â€” a field that no agent generated
  - L3: Every compliance audit for GDPR/HIPAA/CCPA jurisdictions would produce false-positive critical findings for "missing `pii_fields`" when entities correctly use attribute-level `pii: true`
- **Resolution:** `pii_fields` added to spec Section 3 entity-level governance metadata as an optional field with standardised naming. The field is not mandatory in the spec â€” it is available for organisations whose regulatory framework requires an explicit enumerated PII field inventory (e.g., GDPR Article 30, HIPAA Safe Harbor). Agent Artifact understands the field when present. When absent, PII is identified by attribute-level `pii: true` markers. Alternative names (`pii_attributes`, `personal_data_fields`) are prohibited; the canonical name is `pii_fields`.

### CLA-2: Person vs Individual naming inconsistency across spec and example

- **Found by:** Layer 2 (C2), Layer 3 (O5 domain review would catch it; Kenji would flag)
- **Severity:** ~~Critical~~ **Advisory** â€” not a spec violation per owner decision 2026-03-13
- **Evidence:**
  - L2: Spec Section 2 diagram uses `Individual` for natural persons; Financial Crime example uses `Person` â€” different names for the same real-world concept
  - L3: O5 (domain review) would surface this at review time
- **Owner decision:** The spec is naming-agnostic. Users are free to call the entity `Person`, `Individual`, or any other business-aligned name. The requirement is internal consistency within a single domain definition. Using `Person` in the Financial Crime example is valid and intentional. This is not a spec violation. No corrective action required; the finding is downgraded to advisory and closed.

### CLA-3: Product versioning and lifecycle absent from the spec

- **Found by:** Layer 2 (structural weakness, no explicit finding), Layer 3 (D5 scenario scores 1.0/5 â€” complete failure)
- **Severity:** Critical for production use
- **Evidence:**
  - L3: D5 scenario ("version this product that two consumer teams depend on") has no spec-backed answer. Agent Architect can give free-text advice but no structured mechanism.
  - Neither Section 9 (Data Products) nor any agent skill defines product lifecycle states, breaking-vs-non-breaking change classification, or consumer notification mechanism.
- **Recommended fix:** Add product lifecycle states to Section 9 (`status: draft | active | deprecated | retired`), define versioning semantics (major.minor.patch with breaking-change definition), and add a `consumers` or `deprecation_notice` metadata field for communicating changes. Add Product Lifecycle skill to Agent Architect.

### CLA-4: Architecture skill referenced in Agent Architect but does not exist

- **Found by:** Layer 2 (F5), Layer 3 (any architecture philosophy request would fail)
- **Severity:** ~~High~~ **Partially resolved** â€” fallback added 2026-03-13
- **Evidence:**
  - L2: `agents/agent-architect/AGENT.md` line 46 references `skills/architecture/SKILL.md *(planned)*` â€” a non-existent file
  - L3: Any request to "compare MD-DDL to Data Mesh" or "position for CIO" routes to a skill that cannot be loaded. No fallback behaviour defined.
- **Resolution:** Inline fallback protocol added to `agents/agent-architect/AGENT.md`. When the skill file is absent, Agent Architect reads `references/architecture/` and `md-ddl-specification/1-Foundation.md` directly and operates as an informed discussion partner. Full resolution requires implementing `skills/architecture/SKILL.md` per the existing plan at `roadmap/v1/plan-architectureSkill.prompt.md`.

### CLA-5: Cross-agent handoff loses decision context

- **Found by:** Layer 2 (S3), Layer 3 (cross-workflow scores averaged 3.0/5 on Context Continuity)
- **Severity:** ~~Structural weakness~~ **Partially resolved** â€” inline fix applied 2026-03-14; V2 file convention planned
- **Evidence:**
  - L2: Handoff scripts are one-line prompts; hours of Agent Ontology conversation context is lost on handoff to Agent Artifact
  - L3: X2â€“X5 workflows all score 3/5 or lower on Context Continuity. The user carries state between agents manually.
- **V1 resolution (applied):** All sending agents (Ontology, Architect, Governance) now require production of a structured handoff context block before every outbound handoff. All receiving agents (Artifact, Architect, Governance, Ontology) have a startup protocol to read the block when present. Format: Scope, Key decisions, Rejected alternatives, Do not re-open, Task for next agent.
- **V2 resolution (planned):** Durable handoff artifact files (`handoff-to-<agent>.md`) in the domain folder for cross-session handoffs. See `roadmap/v2/plan-handoffArtifact.prompt.md`.

#### L2-C4: Financial Crime diagram shows Party â†” Party self-referential relationship with no definition

~~- Mermaid diagram shows the relationship; Relationships table has no entry; no detail file exists.~~
~~- Fix: Add a self-referential relationship definition. This also creates the spec gap â€” self-referential relationships need explicit spec guidance (which instance is "source", how to express hierarchy depth, how to generate physically).~~

**Resolved 2026-03-14:**

- `Party Related To Party` relationship definition added to [entities/party.md](examples/Financial%20Crime/entities/party.md) with `self_referential: true` and `relationship_attributes` (Association Type, Association Start Date, Association End Date, Verified)

- `self_referential` and `relationship_attributes` fields specified in [5-Relationships.md](md-ddl-specification/5-Relationships.md) with source/target semantics, edge attribute naming, and physical generation guidance (bridge table pattern, adjacency list for hierarchies, recursive CTE recommendation)
- `related_to` relationship type added to the spec's Relationship Types list

#### L2-X2: Event constraint example references forbidden FK attributes

~~- The spec explicitly forbids FK attributes inside entities. The event constraint example uses `Customer.ID` and `Customer Preference.Customer ID` â€” exactly this pattern.~~
~~- Fix: Rewrite the event constraint example using actual entity attributes and relationship navigation.~~

**Resolved 2026-03-14:** Rewrote the `constraints` block in the Event Definition example in [6-Events.md](md-ddl-specification/6-Events.md). Replaced `Ownership Validation` (FK check) with `Preference Must Be Active` (attribute check on `Customer Preference.Status`). Added an inline description explaining that ownership is structural â€” enforced by the `Customer Has Preferences` relationship â€” and that FK-style attribute checks are prohibited.

#### L2-X3: YAML cardinality and Mermaid cardinality have undefined equivalence

- The spec uses English cardinality in YAML (`one-to-many`) and UML notation in Mermaid diagrams (`"1" --> "0..*"`). `one-to-many` could mean `"1..*"` or `"0..*"` on the many side.
- Fix: Define formal cardinality semantics: what does each English form mean for the mandatory/optional multiplicity of both ends?

#### L2-X4: Account entity classDiagram shows many-to-many but relationship YAML defines one-to-many

~~- One-to-many generates a FK on Account. Many-to-many generates a bridge table. These are materially different physical outputs.~~
~~- Fix: Correct the Account classDiagram to show `Customer "1" --> "0..*" Account`.~~

**Resolved 2026-03-14 â€” original fix reversed after domain semantics review:**

The original suggested fix (make the diagram match the one-to-many YAML) was semantically wrong for a Financial Crime domain. Agent Ontology review confirmed that Customer â†” Account is many-to-many: joint accounts, authorized signatories, and beneficial ownership structures all require multiple Customers per Account. The `account.md` diagram was correct; the YAML was wrong.

- [customer.md](examples/Financial%20Crime/entities/customer.md) diagram: `Customer "1" --> "0..*" Account` â†’ `Customer "0..*" --> "0..*" Account`
- [customer.md](examples/Financial%20Crime/entities/customer.md) YAML: `cardinality: one-to-many` â†’ `many-to-many`; `relationship_attributes` added (Holder Type, Holder Start Date, Holder End Date, Is Primary Holder)
- Layer 1 review prompt updated: cardinality conflicts now flag for Agent Ontology domain semantics review rather than recommending a specific correction

#### L2-M1: Polymorphic relationships cannot be expressed

- Common pattern in document management, audit trails, commenting systems. No union/polymorphic target syntax (e.g., `target: Party | Transaction | Account`).

#### L2-M3: Conditional constraints have no first-class syntax

- The `check` constraint accepts expression strings, but no `when` clause on `not_null` and no `IMPLIES` operator in the expression language.

#### L2-M4: Multi-tenancy and tenant-specific model variations cannot be expressed

- No extension or override mechanism beyond inheritance. Inheritance is entity-level, not instance-level.

#### L2-M5: Schema evolution tracking beyond version bumps is absent

- No changelog format, no attribute-rename tracking, no migration path syntax for breaking changes.

#### L2-M6: Cross-entity derived attributes cannot be expressed

- `derived` constraint operates within a single entity. No syntax for `Customer.Total Transaction Value = SUM(Transaction.Amount WHERE linked to this Customer)`.

#### L2-S1: Pre-flight check scope creep has no structural prevention mechanism

- "No additional checks without a spec version bump" is text-only. A contributor adding a sixth check would encounter no resistance.

#### L2-S2: Vocabulary deviation feedback loop does not exist

- `phi` instead of `pii`, `data_class` instead of `classification` â€” agents note these as "potential spec vocabulary gaps" but no aggregation mechanism captures them for spec evolution.

#### L2-S4: Adding new entities to a governed domain creates a perverse maturity regression incentive

- Domain maturity = lowest level of all entities. Adding one ungoverned entity to a Level 3 domain makes it "incomplete at Level 3" until governance is added to the new entity.

#### L2-S5: No cross-domain entity name collision detection

- Consumer-aligned products can reference `Customer` from their own domain and `Customer` from a cross-domain reference. No disambiguation syntax.

#### L2-G1â€“G5: Cross-file consistency checking and diagram semantic correctness absent from review protocols

- See Layer 2 Evaluation Methodology Gaps section. Cross-referencing YAML values against descriptions, relationship YAML against classDiagram cardinality, and agent skill field checklists against the spec schema are not performed by any current review prompt.

### From Layer 3 Only

#### L3-D4: Three-domain product governance conflicts unresolved

- When a consumer-aligned product spans three domains with different retention, classification, and PII postures, the spec's "apply the more conservative requirement" rule is insufficient guidance. Multi-domain masking strategy conflicts have no resolution protocol.

#### L3-R4: Healthcare PHI/PII vocabulary gap

- HIPAA PHI (protected health information) is a different legal concept from general PII. The spec's `pii` boolean does not capture the 18 HIPAA identifiers list, Safe Harbor vs Expert Determination de-identification method, or the PHI minimum-necessary standard. Healthcare practitioners would need vocabulary extensions beyond the current governance schema.

#### L3-O4: FHIR polymorphic references cannot be expressed

- FHIR resources use polymorphic references (a Condition can reference a `Patient | Group | RelatedPerson`). MD-DDL has no union type in relationship targets. Healthcare practitioners modelling FHIR-aligned domains hit this immediately.

#### L3-Industry: ACORD (insurance standard) not referenced anywhere in spec, agents, or skills

- Insurance is a major data management vertical. ACORD is the dominant data standard. No mention anywhere in the spec or agent ecosystem. Insurance practitioners would discover this absence during their first standards-alignment session.

#### L3-O6: No downstream impact assessment when extending an existing domain

- Adding new entities to a production domain has no guidance on which data products reference the affected entities and may need governance, masking, or schema updates.

### Layer 2 â€” Adversarial Review Findings

#### Critical Findings

**C1: `pii_fields` used in agent prompts but absent from spec**

- **Severity:** Critical
- **Files:** `agents/agent-governance/skills/compliance-audit/SKILL.md` lines 165, 189, 282, 350, 365, 374; `agents/agent-governance/AGENT.md` line 128; `agents/agent-architect/skills/odps-alignment/SKILL.md` line 226; regulator files `ccpa.md`, `hipaa.md`, `gdpr.md`
- **Description:** The compliance-audit skill requires `pii_fields` as a governance field on entities. The GDPR regulator file states "`pii_fields:` Must enumerate all attributes containing personal data." The HIPAA regulator file requires "`pii_fields` must enumerate attributes containing any of the 18 identifiers." However, the **spec itself** (Section 3, Governance Metadata Schema) does not define `pii_fields` at all. It defines `pii` (boolean) at both domain and entity level, but nowhere mentions `pii_fields`. The Brownfield Retail example uses it, but the Financial Crime example (the quality benchmark) does not. Agent Governance will flag every entity as non-compliant for missing `pii_fields` when auditing against GDPR, HIPAA, or CCPA, even though the spec provides no schema for this field.

**C2: Diagram-to-domain naming inconsistency â€” "Person" vs "Individual"**

- **Description:** The spec's own example diagram in Section 2 uses `Individual` as the entity name specialising Party. The Financial Crime example uses `Person` instead. Both claim BIAN alignment. This creates incompatible graph nodes for the same concept across examples.

**C3: Person entity governance block contains contradictory retention claims**

- **Severity:** Critical
- **Files:** `examples/Financial Crime/entities/person.md` lines 92-110
- **Description:** The Person entity's governance block declares `retention: 10 years` but the `description` field states "Inherits 7-year retention from Party." These are contradictory. **C4: Financial Crime domain diagram shows `Party <--> |related to|Party` but no relationship definition exists**
